# mlir-systolic 愿景与设计目标

> **目的**：说清 mlir-systolic 要做什么、为谁做、在单核与多核/网络场景下的定位，以及共用脉动阵列与自动调优的方向。

---

## 1. 我们究竟要做一个什么样的 mlir-systolic

mlir-systolic 有**双重使命**，二者互补而非二选一：

| 维度 | 内容 |
|------|------|
| **主线（主要参考 AutoSA）** | 对**单个**嵌套 for 循环（单 kernel），用多面体分析 + 空间-时间映射，自动生成**一套**脉动阵列及其 HLS 代码；保证在这一层上高效、可配置、可复现 AutoSA 行为并做改进。 |
| **扩展（面向 PyTorch/网络）** | 对**多个**层（如 2 个或更多卷积层）时，**不**为每层各生成一套脉动阵列，而是设计**一套共用脉动阵列**，把多层**映射**到这同一套硬件上；再结合自动调优，在资源与性能之间取得可接受的折中。 |

因此可以概括为：

- **单核场景**：参考 AutoSA，多面体驱动的“单嵌套循环 → 单脉动阵列”，追求在该 kernel 上的效率与可配置性。
- **跨层/网络场景**：在单核能力之上，增加“**共用脉动阵列 + 多算子映射**”的能力，并辅以**自动调优**，使整网或子图在单一 FPGA 上可部署、资源可控。

下文分三块展开：单核定位、跨层与共用 SA、自动调优，最后给出对架构与阶段划分的影响。

---

## 2. 单核：主要参考 AutoSA，单循环 → 单脉动阵列

### 2.1 定位

- **输入**：一个满足 SCoP 的嵌套 for 循环（在 MLIR 上即 affine 形式的一个“核”，如单次 GEMM、单次 Conv2D 的稠密部分）。
- **输出**：一套脉动阵列设计 + 对应 HLS C++（PE 阵列、多级 IO、双缓冲、FIFO、pragma 等）。
- **方法**：多面体依赖分析 → 空间/时间划分 → array_part / latency / simd 等分块与标注 → 生成 dataflow 结构与代码；**以 AutoSA 的逻辑与输出结构为主要参考**，在 MLIR 上复现并做可改进点（见 [reference/autosa/AUTOSA_SOURCE_PERF_AND_MLIR_OPPORTUNITIES.md](reference/autosa/AUTOSA_SOURCE_PERF_AND_MLIR_OPPORTUNITIES.md)）。

### 2.2 为什么坚持“单循环一套 SA”在这一层是高效的

- AutoSA 的实践表明：对**单个**嵌套循环，多面体变换能给出规则的数据流、清晰的 PE 与 IO 边界、可预测的 FIFO 与带宽关系，综合后在该 kernel 上往往优于手写或通用 HLS。
- 若在这一层就引入“多 kernel 共用”的约束，会过早限制空间维、tile 形状等，反而可能牺牲单核效率。因此**单核路径应保持“一核一 SA”的生成逻辑**，共用与映射放在上一级处理。

### 2.3 对实现的要求

- 单核 pipeline：Affine → 依赖分析 → space-time 选择 → 分块与 IO/PE/FIFO 生成 → HLS 代码；与 AutoSA 的 stage 对齐（参考 [reference/autosa/AUTOSA_ANALYSIS.md](reference/autosa/AUTOSA_ANALYSIS.md)）。
- 参数化：space_time、array_part、latency、simd 等作为显式配置或属性，便于后续为“共用 SA”和调优提供搜索空间。

## 3. 跨层：多卷积/多算子与“共用脉动阵列”

### 3.1 问题：不能每层一套脉动阵列

- 若对 PyTorch 里每一层（如 Conv1、Conv2、…）都**独立**生成一套脉动阵列：
  - 会得到多套不同的 PE 阵列、多套 IO、多套 FIFO；
  - 在单颗 FPGA 上要么资源爆炸，要么每套都做得很小，利用率与性能都难接受；
  - 层与层之间还要额外的片上/片外衔接，设计复杂且不利于复用。

因此，**不能**简单地把“单核 mlir-systolic”对每一层各跑一遍、得到多套 SA。

### 3.2 目标：一套共用脉动阵列，多层映射上去

- **共用脉动阵列**：在芯片上只实现**一套**物理脉动阵列（例如固定的 PE 规模 P×Q、固定的数据流形态如 output-stationary、固定的 L1/L2/L3 IO 与 FIFO 拓扑）。
- **多层映射**：多个算子（如 2 个或更多卷积层）都**映射到这一套**硬件上运行；每一层对应一种“配置”或“调度”，例如：
  - 该层的循环如何做 tile、哪些维映射到空间/时间；
  - 该层在**同一套 PE 阵列**上以何种 tile 尺寸、何种顺序执行；
  - 可能的不同点：各层的循环界、tile 大小、数据形状，但**PE 阵列尺寸与连接关系不变**。

这样，**硬件只有一份**，通过“每层的映射/配置”在时间上分时执行不同层，或在空间上把不同层映射到同一套数据流上（例如都映射成 2D output-stationary GEMM/conv），从而在资源、带宽和开发复杂度之间取得平衡。

### 3.3 “共用”在编译侧意味着什么

- **阶段一：为“子图/多核”选一个共用 SA 架构**
  - 输入：多个可脉动化的核（如多个 Conv/GEMM 的 Affine 表示）。
  - 输出：**一个** SA 架构描述（如 PE 维度、IO 层级、dataflow 类型、FIFO 深度范围等）。
  - 思路可以是：对每个核先做一次“若单独实现”的 SA 分析，得到各自对 PE 数、带宽、buffer 的需求，再**求交集或选一个可覆盖多核的折中**（例如取各核所需 PE 数的上界或公约形态），或按资源上限做约束求解。

- **阶段二：把每个核映射到这套共用 SA 上**
  - 对每个核：在**固定** SA 形状与数据流的前提下，生成该核的 **tiling / 调度**（即 loop 如何分块、如何对应到 space/time、如何喂入/写出）。
  - 输出：同一份 SA 的 HLS 实现 + 多份“每层配置”（或一个统一调度描述，在运行时按层切换）。

- **与单核 pipeline 的关系**：单核 pipeline 负责“给定一个核，在**无**共用约束时如何生成最优/可选 SA”；共用 SA 的决策与映射则**复用**单核里的分析（依赖、space-time、tile 空间），但增加“架构唯一 + 多核映射”的约束与搜索。

### 3.4 实现路径（建议分阶段）

- **近期**：先把“单核 → 单 SA”做稳、与 AutoSA 对齐，并暴露结构化参数（PE 维、tile 维、space_time 等），为后续共用与调优打基础。
- **中期**：引入“多核描述”（如多个 Affine 核或 linalg op 的集合），实现：
  - **共用 SA 架构选择**：基于多核需求与资源约束，选一个统一的 PE 规模与 dataflow 类型；
  - **每核映射**：在固定 SA 上为每个核生成 tiling 与调度（可视为对单核生成逻辑的约束版：SA 形状固定，只选 tile 与顺序）。
- **长期**：与 PyTorch/ONNX 等前端衔接，把“网络子图 → 多核提取 → 共用 SA + 多核映射”做成端到端流程。

## 4. 自动调优

### 4.1 单核调优（与 AutoSA 类似）

- **搜索空间**：space_time 模式、array_part、latency、simd 等；可选地加上 FIFO 深度、是否做 layout 重排等。
- **目标**：在该 kernel 上最大化吞吐或最小化延迟，或在资源约束下最优。
- **方法**：可用启发式、代价模型或小规模搜索（如 AutoSA 的 tuning 思路）；在 MLIR 侧可把“参数 → 生成 → 评估”做成可脚本化的一环，便于集成到更大的调优框架。

### 4.2 多核/共用 SA 的调优

- **搜索空间**：
  - **共用 SA 架构**：PE 维度 (P, Q)、buffer 大小、dataflow 类型等；
  - **每层映射**：各层在该 SA 上的 tiling、循环顺序等。
- **目标**：整网或子图在给定 FPGA 上的延迟、吞吐或资源利用率（例如多层总时间、或最坏层满足 QoS）。
- **方法**：可先基于代价模型做架构与映射的联合搜索，再对少数候选做 HLS 综合/估算验证；自动调优与“共用 SA + 多核映射”紧密相关，可一起设计接口与数据流。

## 5. 对整体架构与阶段的影响

### 5.1 架构分层（概念上）

```
┌─────────────────────────────────────────────────────────────────┐
│  网络/多核输入 (e.g. PyTorch 子图、多个 Affine 核)                 │
└─────────────────────────────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────────────────────────┐
│  跨层/多核层（可选）                                              │
│  • 共用 SA 架构选择                                              │
│  • 每核映射到该 SA（tiling + 调度）                               │
│  • 自动调优（多核 + 单核参数）                                    │
└─────────────────────────────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────────────────────────┐
│  单核层（核心，主要参考 AutoSA）                                   │
│  • Affine → 依赖分析 → space-time → 分块 → IO/PE/FIFO → HLS     │
│  • 输入：单核 Affine；输出：该核的 SA 设计 + HLS（或“在给定 SA    │
│    上的映射结果”）                                                │
└─────────────────────────────────────────────────────────────────┘
```

- **单核层**：是基础，必须优先做稳、对齐 AutoSA，并保持“一核一 SA”的生成能力。
- **跨层/多核层**：在单核能力之上，增加“共用 SA + 多核映射 + 调优”，面向网络/多算子部署。

### 5.2 阶段建议

| 阶段 | 重点 | 说明 |
|------|------|------|
| **一** | 单核 SA 生成 | 与 AutoSA 对齐并改进（layout/访存、FIFO 深度等）；单核 pipeline 稳定、参数化完整。 |
| **二** | 接口与前端衔接 | 支持从 PyTorch/Allo 等得到 Affine 子图并喂入单核 pipeline；仍可“每子图一 SA”做验证。 |
| **三** | 共用 SA + 多核映射 | 多核输入 → 选一个共用 SA → 每核生成在该 SA 上的映射与配置；输出一份 HLS + 多份配置。 |
| **四** | 自动调优 | 单核参数调优 → 多核/共用 SA 的架构与映射联合调优，与三结合。 |

---

## 6. 小结：用一句话描述 mlir-systolic

- **单核**：在 MLIR 上复现并改进 AutoSA 的“单嵌套循环 → 单脉动阵列”的自动生成，保证单 kernel 上的高效与可配置。
- **跨层/网络**：不 per-layer 各生成一套 SA，而是**设计一套共用脉动阵列**，把多个算子（如多卷积）**映射**到这同一套硬件上，并结合自动调优，使整网或子图可在单 FPGA 上高效、可部署。

两者结合，就是“**以 AutoSA 式单核生成为基础，向上扩展共用脉动阵列与多核映射，并支持自动调优**”的 mlir-systolic。
